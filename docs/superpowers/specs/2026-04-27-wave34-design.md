## Part B вҖ” Subagent Bash-permission fix

### Root cause

- Global `~/.claude/settings.json` runs the orchestrator under `defaultMode: bypassPermissions` with broad `Bash` allow. Main session never sees prompts.
- Project `.claude/settings.local.json` is a narrow exact-string allowlist littered with one-off literal commands like `Bash(echo "2026-04-25T19:00:00Z reading plan")` and `Bash(echo "started hardening")`. Useless for the next subagent вҖ” different timestamp, different agent-id, different exact string.
- Subagents launched via Agent run in a fresh permission context. `bypassPermissions` does not propagate (by design вҖ” it would be unsafe). Subagents fall back to the project allowlist, fail to match any exact string, and either prompt (no human to answer) or block.

### Fix

1. Replace exact-string entries in `.claude/settings.local.json` with curated wildcard patterns the next subagent will actually hit:
   - `Bash(echo *)`, `Bash(date *)`, `Bash(mkdir -p *)`
   - `Bash(git *)`, `Bash(gh *)`
   - `Bash(npm *)`, `Bash(npx *)`
   - `Bash(ls *)`, `Bash(cat *)`, `Bash(grep *)`, `Bash(find *)`
   - Plus any narrowly-scoped commands not safe under wildcards (kept as-is).
2. Add a "Subagent Bash permissions" section to `docs/CLAUDE.md` documenting the convention. **`.claude/` is gitignored project-wide**, so the settings.local.json edit itself does not ship вҖ” it's per-machine. The convention must live in a tracked file so it survives clones, `git clean`, and new contributors.
3. **End-to-end verification.** After B merges, dispatch a throwaway test subagent that runs `git status && npm test --silent --reporter=basic 2>&1 | tail -5` and confirm it does not prompt or hang. The C+A parallel dispatch immediately afterward is the live confirmation.

### Cap

0 src, 0 test, 1 tracked doc file (`docs/CLAUDE.md`). Local-only edit to `.claude/settings.local.json` does not count against the cap (gitignored).

### Done when

- `.claude/settings.local.json` (local) contains only wildcard allow entries; no exact-string leftovers.
- A throwaway dispatch via the Agent tool runs `gh pr list && npm test --silent --reporter=basic` without prompting.
- The "Subagent Bash permissions" section is in `docs/CLAUDE.md` on main.

## Part C вҖ” Storybook dark-mode walk (audit + fix-as-found, capped)

### Scope

Walk **all 40 stories** under `app/src/ui/**/*.stories.tsx` in both light and dark themes via the Storybook theme toolbar wired in W32-B. Note every visual regression. Then **fix the вүӨ8 most-user-visible** regressions in the same PR. File the remainder as backlog rows under a new `### Wave 34 dark-mode follow-ups` heading in `docs/BACKLOG.md`.

"Most user-visible" priority order:
1. Top-level pane components shown on app load (`AppHeader`, primary panels rendered before user interaction).
2. System primitives (`Button`, `Card`, `SectionGroup`, `Field`, `Section`, `EmptyState`, `Badge`).
3. Domain panels gated behind interaction (`SeverityOverridesPanel`, `FindingsPanel`, etc.).

### Cap

вүӨ8 src, вүӨ4 test, вүӨ1 doc (`docs/BACKLOG.md` for the follow-up section).

### Done when

- All 40 stories audited; results recorded in PR description as a table (story вҶ’ light? dark? notes).
- вүӨ8 fixes applied, each addressing a regression listed in the audit table.
- Remaining items appended to `docs/BACKLOG.md` as `[ ]` rows.
- Existing dark-mode regression test (`app/src/test/dark-mode-regression.test.ts`) stays green.
- Storybook still builds cleanly.

### Out of scope

- Chromatic / snapshot infrastructure for stories (separate wave if ever).
- Restructuring stories or adding new variants.
- Token-system changes beyond replacing offending raw colors with existing semantic tokens.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| The wildcard allowlist is *still* not what subagents need (e.g., wrong settings file is consulted, or perms work differently than hypothesized). | B's done-when requires a live throwaway-subagent verification before declaring victory. C+A parallel dispatch is the second confirmation. If both fail, halt and re-investigate before Wave 35. |
| C audit surfaces >8 regressions and the cap forces deferral of something user-visible. | "Most user-visible" priority ordering is defined; if a top-level pane regression is dropped, escalate the cap with explicit approval rather than ship a known visible regression. |

| Stale worktrees from Waves 27вҖ“33 confuse subagent dispatch. | Worktree paths are explicit per part; subagents are pinned to their assigned worktree. Cleanup deferred to its own pass.|
