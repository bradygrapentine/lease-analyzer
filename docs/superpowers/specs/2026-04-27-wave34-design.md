## Part B — Subagent Bash-permission fix

### Root cause

- Global `~/.claude/settings.json` runs the orchestrator under `defaultMode: bypassPermissions` with broad `Bash` allow. Main session never sees prompts.
- Project `.claude/settings.local.json` is a narrow exact-string allowlist littered with one-off literal commands like `Bash(echo "2026-04-25T19:00:00Z reading plan")` and `Bash(echo "started hardening")`. Useless for the next subagent — different timestamp, different agent-id, different exact string.
- Subagents launched via Agent run in a fresh permission context. `bypassPermissions` does not propagate (by design — it would be unsafe). Subagents fall back to the project allowlist, fail to match any exact string, and either prompt (no human to answer) or block.

### Fix

1. Replace exact-string entries in `.claude/settings.local.json` with curated wildcard patterns the next subagent will actually hit:
   - `Bash(echo *)`, `Bash(date *)`, `Bash(mkdir -p *)`
   - `Bash(git *)`, `Bash(gh *)`
   - `Bash(npm *)`, `Bash(npx *)`
   - `Bash(ls *)`, `Bash(cat *)`, `Bash(grep *)`, `Bash(find *)`
   - Plus any narrowly-scoped commands not safe under wildcards (kept as-is).
2. Add a one-line subagent dispatch convention to `.claude/CLAUDE.md`: "subagents inherit `.claude/settings.local.json` allowlist; keep it pattern-based, not exact-string."
3. **End-to-end verification.** After B merges, dispatch a throwaway test subagent that runs `git status && npm test --silent --reporter=basic 2>&1 | tail -5` and confirm it does not prompt or hang. The C+A parallel dispatch immediately afterward is the live confirmation.

### Cap

0 src, 0 test, ≤2 config/docs files (`.claude/settings.local.json`, `.claude/CLAUDE.md`).

### Done when

- `.claude/settings.local.json` contains no exact-string allowlist entries.
- A throwaway dispatch via the Agent tool runs `gh pr list && npm test --silent --reporter=basic` without prompting.
- The convention paragraph is in `.claude/CLAUDE.md`.

## Part C — Storybook dark-mode walk (audit + fix-as-found, capped)

### Scope

Walk **all 40 stories** under `app/src/ui/**/*.stories.tsx` in both light and dark themes via the Storybook theme toolbar wired in W32-B. Note every visual regression. Then **fix the ≤8 most-user-visible** regressions in the same PR. File the remainder as backlog rows under a new `### Wave 34 dark-mode follow-ups` heading in `docs/BACKLOG.md`.

"Most user-visible" priority order:
1. Top-level pane components shown on app load (`AppHeader`, primary panels rendered before user interaction).
2. System primitives (`Button`, `Card`, `SectionGroup`, `Field`, `Section`, `EmptyState`, `Badge`).
3. Domain panels gated behind interaction (`SeverityOverridesPanel`, `FindingsPanel`, etc.).

### Cap

≤8 src, ≤4 test, ≤1 doc (`docs/BACKLOG.md` for the follow-up section).

### Done when

- All 40 stories audited; results recorded in PR description as a table (story → light? dark? notes).
- ≤8 fixes applied, each addressing a regression listed in the audit table.
- Remaining items appended to `docs/BACKLOG.md` as `[ ]` rows.
- Existing dark-mode regression test (`app/src/test/dark-mode-regression.test.ts`) stays green.
- Storybook still builds cleanly.

### Out of scope

- Chromatic / snapshot infrastructure for stories (separate wave if ever).
- Restructuring stories or adding new variants.
- Token-system changes beyond replacing offending raw colors with existing semantic tokens.

### Floor-bump rule (W33 §4-E carry-over)

- Final coverage ≥ 91.2% → bump floor to 91.

- Else no bump; PR description must state the no-bump rationale.

### Done when

- `vitest.config.ts` floor reflects the rule above.
- PR description shows before / after coverage and the chosen floor decision.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| The wildcard allowlist is *still* not what subagents need (e.g., wrong settings file is consulted, or perms work differently than hypothesized). | B's done-when requires a live throwaway-subagent verification before declaring victory. C+A parallel dispatch is the second confirmation. If both fail, halt and re-investigate before Wave 35. |
| C audit surfaces >8 regressions and the cap forces deferral of something user-visible. | "Most user-visible" priority ordering is defined; if a top-level pane regression is dropped, escalate the cap with explicit approval rather than ship a known visible regression. |
| A pushes coverage just under a tier (e.g., 90.69%) and the conditional rule says "no bump." | That's the intended behavior — no contrived defensive-code tests just to clear a tier. PR documents the no-bump rationale. |
| Stale worktrees from Waves 27–33 confuse subagent dispatch. | Worktree paths are explicit per part; subagents are pinned to their assigned worktree. Cleanup deferred to its own pass.|
